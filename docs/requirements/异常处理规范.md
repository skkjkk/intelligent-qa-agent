## 1. 异常处理原则

### 1.1 核心原则
1. **统一处理**：所有异常统一捕获和处理
2. **分层处理**：不同层次异常分别处理
3. **友好提示**：用户看到的错误信息清晰友好
4. **详细日志**：系统日志记录完整异常信息
5. **快速定位**：错误码体系便于问题定位

### 1.2 处理流程
```
异常发生 → 捕获异常 → 分类处理 → 记录日志 → 返回响应
```

## 2. 异常分类

### 2.1 业务异常（BusinessException）
**定义**：业务逻辑层面的异常，可预期的错误

**示例**：
- 用户名已存在
- 密码错误
- 权限不足
- 参数校验失败

**处理方式**：
- 返回业务错误码
- 提示用户友好的错误信息
- 记录INFO级别日志

---

### 2.2 系统异常（SystemException）
**定义**：系统层面的异常，不可预期的错误

**示例**：
- 数据库连接失败
- Redis连接失败
- 文件读写失败
- 内存溢出

**处理方式**：
- 返回系统错误码（500）
- 提示通用错误信息
- 记录ERROR级别日志
- 触发告警

---

### 2.3 第三方异常（ThirdPartyException）
**定义**：调用第三方服务产生的异常

**示例**：
- DeepSeek API调用失败
- 天气API超时
- 网络连接失败

**处理方式**：
- 实现重试机制（最多3次）
- 提供降级方案
- 记录WARN级别日志
- 返回友好提示

## 3. 错误码体系

### 3.1 错误码结构
```
错误码格式：XYZZ
- X: 错误类型（1位）
- Y: 模块编号（1位）
- ZZ: 具体错误（2位）
```

### 3.2 错误类型定义
- **1xxx**：认证授权相关
- **2xxx**：对话相关
- **3xxx**：工具相关
- **4xxx**：工作流相关
- **5xxx**：系统错误
- **9xxx**：其他错误

## 5. 异常处理实现

### 5.1 全局异常处理器

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusinessException(BusinessException e) {
        log.info("业务异常: {}", e.getMessage());
        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)
            .body(new ErrorResponse(e.getCode(), e.getMessage()));
    }

    @ExceptionHandler(SystemException.class)
    public ResponseEntity<ErrorResponse> handleSystemException(SystemException e) {
        log.error("系统异常: ", e);
        return ResponseEntity
            .status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse(5003, "系统内部错误"));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleException(Exception e) {
        log.error("未知异常: ", e);
        return ResponseEntity
            .status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse(5003, "系统内部错误"));
    }
}
```

### 5.2 自定义异常类

```java
public class BusinessException extends RuntimeException {
    private final int code;

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public int getCode() {
        return code;
    }
}
```

## 6. 日志规范

### 6.1 日志级别使用

| 级别 | 使用场景 | 示例 |
|------|----------|------|
| ERROR | 系统错误、严重异常 | 数据库连接失败 |
| WARN | 可恢复的异常、降级 | API调用失败但有降级方案 |
| INFO | 业务异常、关键操作 | 用户登录、业务异常 |
| DEBUG | 调试信息 | 方法入参、中间结果 |

### 6.2 日志格式

```
[时间] [级别] [线程] [类名] - 日志内容
[traceId] [userId] [操作]

示例：
[2026-03-19 10:00:00] [ERROR] [http-nio-8080-exec-1] [UserService] - 用户登录失败
[trace-123] [user-456] [LOGIN_FAILED] username=jujiu, reason=密码错误
```

## 7. 异常处理最佳实践

### 7.1 Controller层
- 只捕获必要的异常
- 不要吞掉异常
- 返回统一的响应格式

### 7.2 Service层
- 抛出业务异常
- 记录关键日志
- 不要捕获系统异常

### 7.3 DAO层
- 让异常向上传播
- 不要捕获数据库异常

## 8. 降级策略

### 8.1 API调用降级
```java
try {
    return deepSeekApi.chat(message);
} catch (Exception e) {
    log.warn("DeepSeek API调用失败，使用降级方案", e);
    return "抱歉，AI服务暂时不可用，请稍后再试";
}
```

### 8.2 缓存降级
```java
try {
    return redisTemplate.opsForValue().get(key);
} catch (Exception e) {
    log.warn("Redis读取失败，从数据库查询", e);
    return database.query(key);
}
```
