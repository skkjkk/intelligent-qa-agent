## 1. 架构概述

### 1.1 架构目标
- **高性能**：响应时间<2秒，支持100+并发
- **可扩展**：模块化设计，易于扩展新功能
- **易维护**：清晰的分层架构，代码规范统一
- **高可用**：故障自动恢复，系统可用性≥99%

### 1.2 架构原则
- 单一职责：每个模块职责明确
- 开闭原则：对扩展开放，对修改关闭
- 依赖倒置：依赖抽象而非具体实现
- 接口隔离：接口设计精简

## 3. 核心模块设计

### 3.1 认证授权模块（Auth Module）

**职责**：
- 用户注册/登录
- JWT Token生成和验证
- 登录日志记录
- 登录失败次数限制（5次/10分钟）
- Token黑名单机制

**核心类**：
```
controller/AuthController            # 认证接口（/api/v1/auth/*）
service/AuthService                # 认证业务逻辑接口
service/impl/AuthServiceImpl       # 认证业务逻辑实现
security/JwtTokenProvider          # JWT Token生成和验证
security/JwtAuthenticationFilter  # JWT认证过滤器（OncePerRequestFilter）
repository/UserRepository          # 用户数据访问（MyBatis-Plus）
repository/LoginLogRepository      # 登录日志数据访问
model/dto/request/LoginRequest     # 登录请求DTO
model/dto/request/RegisterRequest # 注册请求DTO
model/dto/request/RefreshRequest  # Token刷新请求DTO
model/dto/request/LogoutRequest   # 登出请求DTO
model/dto/request/ChangePasswordRequest # 修改密码请求DTO
model/dto/response/LoginResponse  # 登录响应DTO
model/dto/response/UserInfoResponse # 用户信息响应DTO
common/result/ResultCode           # 统一响应状态码枚举
common/exception/BusinessException # 业务异常类
common/exception/GlobalExceptionHandler # 全局异常处理器
```

**Redis使用**：
- `login:fail:{username}` - 登录失败次数计数（5次/10分钟锁定）
- `token:blacklist:{token}` - Token黑名单（单Token失效）
- `user:logout:{userId}` - 用户级Token失效标记（修改密码时使所有Token失效）

### 3.3 DeepSeek AI集成模块（AI Module）

**职责**：
- DeepSeek API调用（非流式）
- DeepSeek API流式调用（WebClient + Flux）
- 提示词管理（System Prompt）
- Token用量统计
- 响应超时控制

**核心类**：
```
client/DeepSeekClient                # DeepSeek API客户端（含流式/非流式方法）
client/DeepSeekResult                # AI响应结果包装类（含Token统计）
config/DeepSeekProperties           # DeepSeek配置（API Key、Model、温度等）
config/WebClientConfig               # WebClient.Builder配置（连接/读取超时）
model/dto/deepseek/DeepSeekMessage  # 消息格式（role、content）
model/dto/deepseek/DeepSeekRequest  # 请求格式（model、messages、stream）
model/dto/deepseek/DeepSeekResponse # 响应格式（choices、usage）
```

**技术实现**：
- 非流式：使用 RestTemplate 同步调用
- 流式：使用 WebClient + Flux 响应式调用，通过 SseEmitter 推送
- 超时配置：连接3秒，读取30秒
- 限流：每用户每分钟最多10条消息

## 7. 安全架构

### 7.1 认证流程

```
用户登录
  ↓
验证用户名密码
  ↓
生成JWT Token
  ↓
返回Token给客户端
  ↓
客户端携带Token访问
  ↓
服务端验证Token
  ↓
允许/拒绝访问
```

### 7.2 安全措施

- **传输安全**：HTTPS加密
- **存储安全**：密码BCrypt加密
- **接口安全**：JWT Token认证
- **防护措施**：限流、防刷、SQL注入防护

### 9.2 推荐配置（Spring Boot Actuator 增强）

**pom.xml 添加依赖**：
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

**application.yml 配置**：
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,env,loggers
  endpoint:
    health:
      show-details: always
  metrics:
    tags:
      application: ${spring.application.name}
```

**暴露的端点**：

| 端点 | 路径 | 说明 |
|------|------|------|
| 健康检查 | `/actuator/health` | 检查 MySQL、Redis 连接状态 |
| 系统信息 | `/actuator/info` | 应用版本、Git信息 |
| 指标数据 | `/actuator/metrics` | QPS、内存、线程等 |
| 日志配置 | `/actuator/loggers` | 动态调整日志级别 |

### 9.3 业务指标监控

**建议自定义指标**：

| 指标名称 | 说明 | 监控意义 |
|----------|------|----------|
| `chat.message.count` | 消息发送次数 | 用户活跃度 |
| `chat.message.success` | 消息成功次数 | 服务稳定性 |
| `chat.message.fail` | 消息失败次数 | 异常告警 |
| `deepseek.api.call` | DeepSeek API 调用次数 | API 成本控制 |
| `deepseek.tokens.usage` | Token 消耗总量 | 成本分析 |
| `auth.login.count` | 登录次数 | 用户活跃度 |
| `auth.login.fail` | 登录失败次数 | 安全监控 |

**实现示例**：
```java
import io.micrometer.core.instrument.Counter;
import io.micrometer.core.instrument.MeterRegistry;

@Service
public class ChatServiceImpl {
    @Autowired
    private MeterRegistry meterRegistry;

    public void sendMessage() {
        // 记录消息发送
        Counter.builder("chat.message.count")
            .tag("userId", userId.toString())
            .register(meterRegistry)
            .increment();
    }
}
```

### 9.4 日志规范

**日志级别使用**：

| 级别 | 使用场景 |
|------|----------|
| ERROR | 系统错误、异常堆栈 |
| WARN | 可恢复的异常、降级处理 |
| INFO | 业务流程关键节点（登录、发送消息） |
| DEBUG | 方法入参、中间变量 |

**日志格式**：
```
[时间] [级别] [线程] [类名] - [日志内容]
[2026-03-24 10:00:00] [INFO] [http-nio-8080-exec-1] [AuthService] - [AUTH][LOGIN_SUCCESS] userId=1, username=jujiu
```

**日志分色配置**（Logback）：
```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%highlight(%-5level)] [%cyan(%thread)] [%cyan(%logger{36})] - %msg%n</pattern>
```

### 9.5 可选进阶方案

#### 方案一：Prometheus + Grafana（中等复杂度）

| 组件 | 作用 |
|------|------|
| Prometheus | 采集和存储时序指标 |
| Grafana | 可视化仪表盘 |

**架构**：
```
应用 → Micrometer → Prometheus → Grafana
```

**适用场景**：需要长期监控和可视化展示

---

#### 方案二：ELK 日志聚合（中等复杂度）

| 组件 | 作用 |
|------|------|
| Elasticsearch | 日志存储和检索 |
| Logstash | 日志收集和过滤 |
| Kibana | 日志可视化 |

**架构**：
```
应用 → Logstash → Elasticsearch ← Kibana
```

**适用场景**：多实例日志统一查询

---

#### 方案三：SkyWalking 链路追踪（高复杂度）

**作用**：
- 分布式请求链路追踪
- 性能瓶颈分析
- 错误定位

**适用场景**：微服务架构、生产环境
